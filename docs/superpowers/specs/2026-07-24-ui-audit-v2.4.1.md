## 二、逐维度深度审计

### 2.1 设计 Token 体系

**现状**：
- `tokens.css`（88 行）定义了间距、圆角、字体、阴影、过渡、语义色、组件 Token
- `themes.css`（4456 行）在 `:root` 中**重复定义**了字体、间距、语义色、徽章色
- 变量命名不统一：`--sp-1` vs `--radius-sm` vs `--font-xs` vs `--fs-base`（tokens.css 和 themes.css 中的 `--font-*` 前缀不一致）
- tokens.css 中 `--fs-*` 系列未定义，但 layout.css 引用了 `--fs-2xl`、`--fs-base` 等

**问题**：
1. 同一变量两处定义，修改时易遗漏
2. 前缀体系混乱（`--sp-` / `--font-` / `--fs-` / `--fw-` / `--r-` / `--radius-`）
3. tokens.css 的 `--font-sans` 与 themes.css 的 `--font-sans` 完全相同但定义两次
4. 缺少 `--fs-*` 字体大小 Token 的集中定义

**建议**：
- 统一 Token 前缀：`--space-*` / `--radius-*` / `--font-size-*` / `--font-weight-*` / `--color-*`
- 所有 Token 集中到 tokens.css，themes.css 只定义主题色变量
- 删除 themes.css 中的重复定义

### 2.2 字体系统

**现状**：
- 字体栈：`-apple-system, BlinkMacSystemFont, 'Segoe UI', 'PingFang SC', 'Hiragino Sans GB', 'Microsoft YaHei'`
- 字号层级：11px / 12px / 14px / 16px / 18px / 22px / 26px / 36px（Major Third 1.25 比例）
- 字重：400 / 500 / 600 / 700
- 行高：1.2 / 1.5 / 1.8

**问题**：
1. 中文字体栈缺少 `'Noto Sans SC'`（开源高质量中文字体）和 `'HarmonyOS Sans'`（鸿蒙系统字体）
2. 等宽字体栈缺少 `'Cascadia Code'`（Windows Terminal 默认）
3. 字号变量使用率低：themes.css 中仍有 303 处 `font-size:` 硬编码
4. 核心数据展示（stat-value）字号 26px 偏小，金融数据面板应更大
5. 数字（股票代码、金额、涨跌幅）未使用等宽/tabular-nums，导致对齐不整齐

**建议**：
- 所有 `font-size:` 替换为 CSS 变量引用
- 数字列使用 `font-variant-numeric: tabular-nums`
- 金融数据面板字号提升至 32px+
- 补充中文字体 fallback

### 2.3 图标系统

**现状**：
- 4 套图标系统：emoji / ink / edge / crystal
- emoji 使用 Unicode 字符（📈🗓️🤖🔬⚙️）
- ink 使用中文单字（策历智研设）
- edge 使用 SVG inline stroke 风格
- crystal 使用 SVG inline fill 风格
- 切换通过 `icon-system-*` CSS class 控制图标容器样式

**问题**：
1. **ink 图标**：中文单字在 56px 侧边栏中勉强可读，但在 36px 容器（CSS 定义）中缺少视觉重量
2. **edge/crystal SVG**：`currentColor` 继承，在暗色主题下可能因 inherit 链断裂而不可见
3. 图标切换仅影响侧边栏导航，卡片内 emoji（如 📅📈🎯💎）不受切换影响
4. 缺少统一的图标尺寸规范（nav-icon 18px，mobile-nav-icon 单独定义）
5. 没有提供图标预览或切换时的视觉反馈

**建议**：
- 统一所有 emoji 到图标系统（包括卡片内和页面标题）
- 为 ink 风格增加圆形背景提升视觉重量
- edge/crystal SVG 添加 `stroke="currentColor"` 和 `fill="currentColor"` 显式声明
- 增加图标切换过渡动画

### 2.4 配色与主题

**现状**：
- 7 套主题，每套定义 30+ CSS 变量
- 默认主题 tech-blue（科技蓝），金融风格
- dark-pro 主题完整覆盖 Element Plus 组件（按钮、表格、对话框、输入框、标签页、分页、消息框、下拉菜单、标签、加载遮罩）
- 每套主题对 Element Plus 按钮有 3 层覆盖（default / hover / active / plain / text）

**问题**：
1. **classic 系列主题冗余**：classic-white / classic-red / classic-gold 三套本质相同（白底+不同主色），可以用 CSS 变量驱动合并为 1 套 + 2 个色相变量
2. **Element Plus 覆盖代码膨胀**：每套主题重复 30-50 行 `.el-button--primary` 等覆盖代码，7 套主题 × 30 行 ≈ 210 行重复
3. **暗色主题对比度**：部分组件（如 el-tag）在 dark-pro 下边框色 `#3a4a6a` 与背景色 `#233554` 对比度仅 1.5:1，低于 WCAG AA 标准
4. 主题切换无过渡动画，色彩突变生硬
5. 用户菜单中主题选择器使用 emoji 色块，色块太小（12px）不易辨识

**建议**：
- 合并 classic 系列为参数化主题
- Element Plus 覆盖使用 CSS 变量继承，减少重复
- 为主题切换添加 `transition: background-color 0.3s, color 0.3s`
- 主题选择器改为大色块预览

### 2.5 布局系统

**现状**：
- 桌面端：220px 固定侧边栏 + 自适应内容区
- 平板端（768-1024px）：侧边栏自动折叠为 56px
- 移动端（<768px）：侧边栏隐藏，底部导航栏
- 卡片网格：4 列 / 3 列 / 2 列 / 1 列（响应式）
- 全局 Header：二级导航标签 + 搜索框 + 日期选择器 + 用户菜单

**问题**：
1. **平板端侧边栏逻辑缺陷**：`responsive.css` 中 `.sidebar:not(.collapsed)` 强制设为 56px，但 `.sidebar .nav-item span` 隐藏所有 span（包括图标），导致平板端侧边栏完全空白
2. **底部导航栏**：仅显示 emoji 无文字标签，新用户无法理解功能
3. **内容区最大宽度**：桌面端无 `max-width` 限制，在超宽屏上内容拉伸过度
4. **全局 Header 布局**：搜索框、日期选择器、用户菜单挤在同一行，小屏时布局混乱
5. **卡片间距**：`margin-bottom: 20px`（硬编码）与 `gap: var(--sp-4)`（16px）不一致

**建议**：
- 修复平板端侧边栏图标显示
- 底部导航栏添加文字标签
- 内容区添加 `max-width: 1400px` + `margin: 0 auto`
- 全局 Header 重构为 flex-wrap 三区布局
- 统一卡片间距为 CSS 变量

### 2.6 交互体验

**现状**：
- 页面切换使用 `<Transition name="fade-slide" mode="out-in">`（5 处）
- 分数变化有数字滚动动画
- 侧边栏折叠有 localStorage 持久化
- 主题切换即时生效
- 美林时钟阶段切换有通知

**问题**：
1. **无键盘导航**：侧边栏菜单不支持 Tab 键聚焦，`Enter` 键无法激活
2. **无面包屑导航**：深层次页面（如 策略研究 → 回测历史）无法快速返回
3. **无加载骨架屏**：animations.css 定义了 skeleton 类但模板中未使用
4. **无操作反馈**：删除、登出等危险操作无确认对话框
5. **无页面滚动位置记忆**：切换页面后返回，滚动位置丢失
6. **搜索框**：`trigger-on-focus="false"` 需要手动输入才触发，体验不友好
7. **表格排序**：无点击列头排序功能
8. **数据刷新**：仅日历页有手动刷新按钮，其他页面无刷新机制

**建议**：
- 侧边栏菜单添加 `tabindex` 和 `@keydown.enter` 支持
- 添加面包屑组件
- 在数据加载中使用骨架屏替代空白 loading
- 危险操作添加 `ElMessageBox.confirm`
- 使用 `<keep-alive>` 缓存页面状态
- 表格列头可排序

### 2.7 代码架构

**现状**：
- index.html 7032 行单体文件，包含模板 + 全部 JS 逻辑
- 8 个 Composables 文件（useAuth / useTheme / useNavigation / useCalendar / useDashboard / useAI / useWatchlist / useSystem）
- 独立的 JS 模块（themes.js / icons.js / core.js / charts.js / calendar.js / strategies.js / ai.js / system.js / users.js）
- 所有 JS 通过 IIFE 挂载到 `window.__quantModules`

**问题**：
1. **8 个 Composables 是死代码**：定义了但 setup() 中重复实现了相同逻辑，没有使用 `init()` 返回的对象
2. **setup() 函数过于庞大**：从 2634 行到 7027 行，约 4400 行逻辑全部在一个函数中
3. **全局变量污染**：800+ 个变量在 setup() 闭包中，调试困难
4. **Composables 与独立 JS 模块功能重叠**：themes.js 和 useTheme.js 都定义了主题切换逻辑
5. **无 TypeScript 或 JSDoc 类型标注**：所有参数无类型约束
6. **CSS 层级深度**：themes.css 中嵌套最深的选择器达 5 层（如 `[data-theme="dark-pro"] .el-table--striped .el-table__body tr.el-table__row--striped td.el-table__cell`）

**建议**：
- 将 setup() 中的逻辑按功能拆分到 Composables，并实际使用它们
- 删除重复的独立 JS 模块（themes.js 合并到 useTheme.js）
- 添加 JSDoc 类型注释
- 减少 CSS 选择器嵌套深度

### 2.8 移动端体验

**现状**：
- responsive.css 定义了 Mobile（<768px）和 Small Mobile（<480px）两档
- 移动端侧边栏完全隐藏，改用底部导航栏
- 卡片网格变为单列
- 对话框宽度 96%

**问题**：
1. **底部导航栏**：只有 emoji 图标，无文字标签，可发现性差
2. **无手势支持**：无左右滑动切换页面
3. **表格体验差**：移动端无横向滚动指示器，用户不知道可以左右滑动
4. **日期选择器**：移动端原生 datepicker 弹出体验优于 Element Plus 的
5. **安全区域**：底部导航栏使用了 `env(safe-area-inset-bottom)` 但顶部状态栏未适配

**建议**：
- 底部导航栏添加文字标签
- 表格添加 `-webkit-overflow-scrolling: touch` 和滚动阴影指示器
- 添加触摸滑动切换页面手势
- 顶部添加 `padding-top: env(safe-area-inset-top)`

**结论**：当前 UI/UX 处于 **"功能可用但体验粗糙"** 阶段。核心业务逻辑完整，但设计系统碎片化严重、代码架构债务重、交互细节缺失。建议分两阶段改进：第一阶段修复 P0/P1 问题（2-3 天），第二阶段系统重构架构（1-2 周）。
